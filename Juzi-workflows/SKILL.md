# 工作流组合

> "政策和策略是党的生命，各级领导同志务必充分注意，万万不可粗心大意。"
> —— 毛泽东

方法论的威力不在于单独使用某一件思想武器，而在于**在正确的时机以正确的顺序组合使用**。本 skill 提供四条标准化工作流，定义 skill 的调用顺序、步骤间的数据传递格式和终止条件。

---

## Workflow 1：新项目启动

**适用场景：** 从零开始面对一个新任务/项目/领域，目标已知但路径未知，资源有限。

**触发信号：** 新项目、新领域、MVP、从零开始、不知道从哪里下手

### 工作流

```
investigation-first → contradiction-analysis → spark-prairie-fire → protracted-strategy
     调查研究        →     矛盾分析          →     星火燎原        →     持久战略
```

### 步骤详解

**Step 1：investigation-first（调查研究）**
- 目标：摸清现状，占有第一手材料
- 输出格式（传递给 Step 2）：
  ```
  调查结论：
  - 现状是：……
  - 关键约束是：……
  - 主要的未知项是：……
  - 已知的可依托基础是：……
  ```
- 终止条件：能够回答"现在是什么情况"这个问题

**Step 2：contradiction-analysis（矛盾分析）**
- 输入：Step 1 的调查结论
- 目标：从调查材料中识别主要矛盾，找到最值得突破的切入点
- 输出格式（传递给 Step 3）：
  ```
  主要矛盾：[A] vs [B]
  矛盾性质：[对抗性/非对抗性]
  最值得突破的切入点：……
  原因：解决了它，[X]、[Y] 问题随之缓解
  ```
- 终止条件：找到一个明确的主要矛盾和切入点

**Step 3：spark-prairie-fire（星火燎原）**
- 输入：Step 2 识别的切入点
- 目标：以切入点为根据地，制定从小到大的发展路线
- 输出格式（传递给 Step 4）：
  ```
  根据地：[具体切入点]
  发展路线：
    第1步：[具体行动] → 成功标志：……
    第2步：[具体行动] → 成功标志：……
    第3步：[具体行动] → 成功标志：……
  ```
- 终止条件：有一条清晰的、从当前能力出发的发展路线

**Step 4：protracted-strategy（持久战略）**
- 输入：Step 3 的发展路线
- 目标：将发展路线纳入阶段性战略框架，确定当前处于哪个阶段，设置阶段转换条件
- 输出格式：
  ```
  当前阶段：战略防御期
  本阶段核心任务：[Step 3 第1步]
  转入相持期的条件：当 [具体里程碑] 达成时
  ```
- 终止条件：有明确的阶段定位和转换条件

## Workflow 2：复杂问题攻坚

**适用场景：** 面对一个已知存在但难以解决的具体问题，问题边界清楚但原因不明或路径不清。

**触发信号：** 疑难 bug、架构瓶颈、反复失败的尝试、根因不明的问题

### 工作流

```
investigation-first → contradiction-analysis → concentrate-forces → practice-cognition → criticism-self-criticism
     调查研究        →     矛盾分析          →     集中兵力        →    实践认识论      →     批评与自我批评
```

### 步骤详解

**Step 1：investigation-first（调查研究）**
- 目标：弄清问题的真实面目（而非症状）
- 输出（传递给 Step 2）：
  ```
  调查结论：
  - 问题的现象是：……
  - 问题出现的条件是：……
  - 已经排除的原因是：……
  - 还未调查的方向是：……
  ```
- 终止条件：能够精确描述问题（而非"好像是 X 出了问题"）

**Step 2：contradiction-analysis（矛盾分析）**
- 输入：Step 1 的调查结论
- 目标：找到问题中的主要矛盾（根本冲突点）
- 输出（传递给 Step 3）：
  ```
  主要矛盾：[根本冲突点]
  我的假说：这个冲突导致了……，因为……
  验证方式：如果假说正确，应该能观察到……
  ```
- 终止条件：有一个可以验证的假说

**Step 3：concentrate-forces（集中兵力）**
- 输入：Step 2 的假说和验证方式
- 目标：围绕验证这个假说，集中全部力量，拒绝分心
- 输出（传递给 Step 4）：
  ```
  🎯 主攻目标：验证假说"[假说内容]"
  验证方法：[具体的验证步骤]
  完成信号：[假说被证实或证伪]
  暂缓的其他方向：……
  ```
- 终止条件：假说被验证（证实或证伪均可）

**Step 4：practice-cognition（实践认识论）**
- 输入：Step 3 的验证结果
- 目标：根据验证结果，更新认识，进入下一轮（如果假说被证伪则回到 Step 2）
- 循环终止条件：问题被彻底解决（可以明确描述原因和解决方案）

**Step 5：criticism-self-criticism（批评与自我批评）**
- 输入：整个攻坚过程
- 目标：审视攻坚过程中的方法论失误，防止下次犯同类错误
- 必须输出：结构化工作审视报告

## Workflow 3：方案迭代优化

**适用场景：** 已有一个基础方案，需要通过收集反馈、分析问题、迭代改进来提升质量。

**触发信号：** 迭代优化、用户反馈、代码审查后改进、方案效果不理想

### 工作流

```
mass-line → contradiction-analysis → practice-cognition → criticism-self-criticism → mass-line（循环）
 群众路线 →     矛盾分析          →    实践认识论      →     批评与自我批评      →  群众路线（下一轮）
```

### 步骤详解

**Step 1：mass-line（群众路线）**
- 目标：从多个信息源收集对现有方案的反馈
- 信息源清单（根据具体情况选择）：
  - 用户的明确反馈
  - 测试结果/运行结果
  - 代码审查意见
  - 性能数据/指标
  - 代码库中相关模式的启示
- 输出（传递给 Step 2）：
  ```
  收集到的多源反馈：
  - [信息源1]：……
  - [信息源2]：……
  各源之间的矛盾之处：……
  共同指向的问题：……
  ```
- 终止条件：有来自至少 2 个信息源的反馈

**Step 2：contradiction-analysis（矛盾分析）**
- 输入：Step 1 的多源反馈
- 目标：从反馈中识别出需要优先解决的主要矛盾
- 输出（传递给 Step 3）：
  ```
  本轮迭代的主要矛盾：[当前方案的核心问题]
  改进假说：如果……，则……
  验证方式：……
  ```

**Step 3：practice-cognition（实践认识论）**
- 输入：Step 2 的改进假说
- 目标：实施改进并验证效果
- 循环终止条件（开始前明确说明）：当 [具体指标] 达到 [具体数值/状态] 时，本轮迭代完成

**Step 4：criticism-self-criticism（批评与自我批评）**
- 目标：审视本轮迭代的过程质量
- 必须输出：工作审视报告

**Step 5：mass-line（下一轮）**
- 基于本轮的改进结果，重新收集反馈，开始新的迭代循环

**整体循环终止条件：** 当 mass-line 的多源反馈显示"没有新的重要问题"或用户确认满意时，退出循环。

## Workflow 4：生产变更与事故恢复

**适用场景：** 真实服务、端口、Runtime、系统配置或用户流量可能受影响的部署、重启、切换、修复与事故恢复。

**触发信号：** production、线上、服务重启、停止进程、切换路由、管理员权限、用户已受影响、需要回滚。

### 不可跳过的安全门

**Step 0：动作分级与变更卡**

- 将每个动作标为只读、可逆或破坏性；`stop`、`restart`、`kill` 一律属于破坏性动作。
- 写明目标、用户影响、爆炸半径、受保护资源、单次破坏性动作预算、停止条件、回滚和救援路径。
- 用户明确授权重启不等于授权重复尝试；除非用户明确授权一个系列且工作区规则允许，否则每次授权最多执行一次破坏性请求。

**Step 1：冻结当前可用基线（`investigation-first`）**

至少记录服务状态、父子进程/PID、监听端口、轻量健康 API、近期错误日志和一条真实业务流量证据。用户手动恢复的 Runtime/PID 立即标为受保护基线；可靠替代者形成前，不得停止、清理或覆盖它。

**Step 2：在生产外证明失败安全（`practice-cognition`）**

先在测试、临时目录、隔离进程或 staging 中验证正常路径，并注入 `stop`、`start`、`cleanup`、`wait` 超时、UAC 取消、进程未退出、端口未释放以及日志/统计/诊断失败。控制面必须能在日志或遥测失效时继续启动。未经这些验证的脚本不得首次作用于生产。

**Step 3：准备 standby**

在不释放基线的前提下准备替代者。确认其配置、依赖、进程、端口可用性、健康接口和日志均满足切换条件；无法并存时，也必须先完成离线启动验证和明确的救援命令。

**Step 4：事务化切换**

按“预检 → 单次切换 → 健康确认 → 提交；失败则回滚”的顺序执行。切换期间不并行做日志优化、清理、重构或第二套重门禁。任何破坏性步骤失败后立即停止，保留证据并回到 Step 1，不得自动重复 stop/restart/kill。

**Step 5：真实链路验收**

UAC 窗口消失、回执存在或 exit code `0` 都只证明命令执行过，不证明服务成功。必须从普通权限新进程重新查询：服务状态、父子进程、端口、Runtime API、错误日志，以及适用时的一条真实流量。只有这些证据一致才可宣布切换成功。

**Step 6：回滚与救援**

验证失败时优先恢复最后可用基线，再分析根因。若用户已手动恢复，立即冻结该 Runtime，不再由自动化操作。恢复前不得为“清理干净”而删除现场、杀掉未知进程或继续扩大系统范围。

### 验收成本分级

- **普通代码或行为迭代：** 定向回归、一次完整门禁和几分钟短时冒烟三项缺一不可；纯文档/注释可按项目明确的窄门禁执行。不得每次代码改动都运行长窗口验收，也不得重复跑等价重门禁。
- **重大升级、重大重构或生产控制面变更：** 在普通门禁之上增加 **10 分钟快速验收**，覆盖持续健康、真实流量和资源稳定性。
- **跨天观察：** 作为异步滚动证据，不默认阻塞快速迭代；只有用户或发布契约明确要求时才作为发布门槛。

### 事故模式

服务已经中断时，主要目标立即变为“恢复并保护可用服务”。先止血、保留证据、建立用户影响和停机窗口，再做确定性根因分析。根因尚不明确不妨碍回滚，但不得把推测写成结论。

### 生产变更交接格式

```text
生产变更卡：
- 动作等级：只读 / 可逆 / 破坏性
- 受保护基线：……
- 用户影响与爆炸半径：……
- standby 就绪证据：……
- 单次切换步骤：……
- 停止条件：……
- 回滚 / 救援：……
- 验收等级：普通 / 重大 10 分钟 / 明确要求的跨天观察
- 成功证据：服务、进程、端口、Runtime API、日志、真实流量
```

## 工作流选择指南

| 你的情况 | 推荐工作流 |
|---------|---------|
| 面对全新的项目/任务，不知道从哪里下手 | Workflow 1：新项目启动 |
| 有一个具体的疑难问题，需要找到根因并解决 | Workflow 2：复杂问题攻坚 |
| 已有方案但效果不理想，需要迭代优化 | Workflow 3：方案迭代优化 |
| 可能影响真实服务、系统配置或用户流量 | Workflow 4：生产变更与事故恢复 |
| 以上都不完全符合 | 回到 `arming-thought` 的调度表，单独调用最匹配的 skill |

## 数据传递规则

工作流中每个步骤的**输出必须显式传递给下一步骤**。具体做法：

1. 每个步骤结束时，写出格式化的"传递给下一步的信息"
2. 下一步开始时，确认收到并引用上一步的输出
3. 不得跳步骤执行（如不得跳过 contradiction-analysis 直接进入 concentrate-forces）

## 工作流中断处理

如果在工作流执行过程中遇到阻塞：
- **信息不足** → 插入一次 `investigation-first`，补充信息后继续
- **假说被证伪** → 返回 `contradiction-analysis`，重新分析
- **发现新的更重要的矛盾** → 用 `overall-planning` 评估是否需要调整工作流
- **生产破坏性步骤失败** → 停止重试，保护/恢复最后可用基线，记录证据并返回 Workflow 4 Step 1
