# 调研与核实

## 1. 先锁定时间语义

- 读取当前日期和用户时区，把“本周五”“下个月”等转换成 `YYYY-MM-DD`。
- 检查去程、返程和住宿晚数是否一致。
- 默认值必须显式写入攻略，例如“1 位成人、二等座、两人同住”。

## 2. 来源优先级

按以下顺序检索，并尽量使用能直接支持具体结论的页面：

1. 交通运营方、政府、景区、轮渡、酒店品牌官网。
2. 官方天气与开放数据服务。
3. 权威指南和目的地政府文旅资料。
4. 地图地点页，用于地址、营业位置和导航入口。
5. 商业预订平台和内容网站，仅用于补充价格区间与体验，不作为唯一事实来源。

每条易变事实记录：`verifiedAt`、来源 URL、快照或实时状态、需要用户复核的边界。

## 3. 交通与票务

- 高铁/火车：核对车站、车次、出发到达时间、历时、席别、票价和目标日期。
- 不把一次成功的内部接口调用当成稳定网页能力。12306 等站点若受 CORS、重定向或反爬限制，网页只放预填官方入口。
- 余票写成“核实于某时的快照”，不得承诺未来仍有票。
- 飞机、长途客运、轮渡同样优先运营方官网；注明提前到达、证件、改退和最晚登乘要求。
- 行程连接处做时间算术，预留进站、换乘、寄存、步行和天气缓冲。

## 4. 住宿、景点与餐饮

- 住宿优先固定区域和不换店，给出首选与 2–3 个价位备选；价格写成区间并提示房态实时变化。
- 景点核对开放日、入园时段、预约、门票、停止入场和交通入口。
- 餐饮核对店名、地址、招牌、适合的行程站点和过敏风险；不要堆叠过多同类主食。
- 推荐来源可参考米其林、政府老字号名录或本地权威指南；图片只借鉴卡片层级，不复制第三方图片。

## 5. 天气与地图

- 保存生成时天气快照；联网刷新仅作为增强。
- Open-Meteo 等接口失败时保留快照，并展示最后核实时间和天气官方入口。
- 免 Key 地图 URI 可用于搜索/导航；完整 JS 地图需要 Key 时，改用自包含动线图加外部导航链接。

## 6. 行前与途中执行覆盖矩阵

在结束检索前逐项确认，无法核实时写明缺口和官方复核入口，不用猜测补齐：

- 预订与证件：车票、机票、船票、门票、酒店、实名证件和取消规则。
- 本地移动：到站接驳、每天第一站和最后一站、换乘时间、末班风险、步行强度和行李处理。
- 逐站执行：每站使用 `HH:MM`，包含地址、费用/预约、地图入口、停留时长和至少一个风险提示。
- 天气兜底：高温、降雨、台风、冰雪等对具体站点的影响，以及可执行的室内或低风险替代。
- 支付与通信：仅在目的地确有差异时说明支付方式、网络、插座、漫游或离线凭证。
- 健康与安全：按人群和目的地补充过敏、药物、无障碍、保险、紧急电话或高海拔等真实相关事项。

## 7. 调研停止条件

满足以下条件即可进入写作，不为“更多推荐”无限扩张：

- 往返交通与关键接驳可闭环。
- 每天有主线、备选和天气兜底。
- 酒店、船票/门票、美食、预算和行李均有可执行信息。
- 所有易变事实有核实时间或明确的官方复核入口。
- 来源能直接支持攻略中的关键结论。
