# 旅游规划产品竞品学习与 Juzi 增量路线

## 1. 研究目的

本报告研究普通用户规划 `2-7 天自由行` 时会使用的旅游规划产品，重点回答四个问题：

1. 页面如何组织“灵感、决策、预订、执行、复盘”；
2. 哪些模块是真正的主流程，哪些只是营销或交易入口；
3. 产品需要哪些数据，以及数据多久刷新、失效后如何兜底；
4. 哪些交互适合增量加入 `Juzi-travel-guide`，哪些会破坏当前原版。

研究执行时间为 `2026-07-16` 至 `2026-07-17`。研究对象以公开官网和无需登录的真实页面为主，没有注册、登录、提交规划表单或进行预订。

### 证据标记

- **[声明]**：官网、帮助中心或产品页面明确写出的能力。
- **[观察]**：本次在真实页面、首屏截图或可访问 DOM 中直接看到的布局与交互。
- **[推断]**：基于前两类证据得出的产品设计结论，不冒充产品事实。

## 2. 执行摘要

旅游规划网站没有一种万能布局，它们大致分成四类：

1. **行程编辑器型**：Wanderlog、Tripomatic 把“逐日行程 + 地图”作为核心工作区，适合反复增删和调整顺序。
2. **预订归档型**：TripIt 先解决确认邮件、航班和酒店信息散落的问题，再提供提醒和途中指导。
3. **垂直搜索型**：Google Travel 把结构化输入、价格比较、趋势和提醒做深，不承担完整攻略叙事。
4. **内容与交易型**：穷游和携程用社区行程、点评、游记、问答降低决策成本，再连接酒店、机票、门票等交易入口。

`Juzi-travel-guide` 当前更接近第五类：**专家策划后交付的可执行单文件攻略**。它已有强叙事首屏、明确车次、逐时行程、路线图、船票、酒店、小吃、预算、天气刷新、备忘和来源，不应被改造成需要登录的在线编辑器。

最值得吸收的不是竞品的外观，而是三条主线：

- 用 Wanderlog/Tripomatic 的“行程与地图同步”降低空间认知成本；
- 用 TripIt/Google Travel 的“状态、时效、提醒和失败兜底”提高途中可信度；
- 用穷游/携程的“模板与内容证据”降低冷启动，但必须过滤过期、重复和低质量信息。

下一轮应继续基于厦门原版增量迭代。`P0` 先补信息状态与移动执行效率，`P1` 再补地图联动和轻量比较，`P2` 才考虑可编辑排序、协作或复杂 AI 规划。

## 4. 四种高频布局模式

### 4.1 行程与地图双工作区

**代表**：Wanderlog、Tripomatic、Roadtrippers。

- **[观察]** Wanderlog 首屏产品图把行程卡片放左侧、地图和路线放右侧，用户无需在日程和地图 App 之间来回切换。
- **[声明]** Wanderlog 支持查看地点间距离与时间、指定访问时间和路线优化；Tripomatic 支持逐日拖拽和多模式路线。
- **[推断]** 桌面端适合 `行程 60% + 地图 40%`；移动端不应硬塞双栏，应该改为“行程 / 地图”分段切换，并保留当前选中日期和站点。

适合 Juzi 的方式不是重做页面，而是在原版“逐日行程”和“路线地图”之间建立联动：点击某一行程站点时，地图切换到对应日期并高亮该点；从地图返回时滚动到对应时间节点。

### 4.2 自动归档与下一步

**代表**：TripIt。

- **[声明]** 用户把任意预订确认邮件转发到 TripIt，系统自动建立综合行程；Pro 提供航班变化和提醒。
- **[观察]** 首页没有先展示景点瀑布流，而是按“自动整理 -> 实时提醒 -> 每站指导”讲完整任务链。
- **[推断]** 对旅行中的用户，“下一步做什么、凭证在哪里、是否有变化”比更多灵感更重要。

Juzi 不需要接管邮箱。更合适的实现是保留本地单文件：把车次、船票、酒店、预约状态统一成“行程凭证区”，支持用户在本机标记已订、存备注和打开官方订单入口。

### 4.3 结构化搜索与比较

**代表**：Google Travel。

- **[观察]** 搜索表单一次收集往返类型、人数、舱位、出发地、目的地和日期；结果工具强调日期网格、价格图、历史趋势和价格跟踪。
- **[观察]** 页脚明确写出价格、供应和行程详情来自合作伙伴的最新信息，反映 `24 小时以内` 的结果，并要求预订前向服务商复核。
- **[推断]** 易变数据不能只给一个看似精确的数字；必须同时给出范围、口径、币种、核实时间、限制条件和复核入口。

Juzi 的交通、住宿和预算模块应优先增加“比较状态”和“为什么推荐”，而不是内嵌不可控的私有票务接口。

### 4.4 模板、口碑与交易

**代表**：穷游、携程、Tripadvisor。

- **[观察]** 穷游把“创建新行程”和公共行程列表放在同一页面，支持按目的地和热度/复制量筛选。
- **[观察]** 携程攻略页把点评、游记、问答、目的地目录和酒店/机票/门票等入口放在同一框架中。
- **[观察]** Tripadvisor Trips 的登录前空状态给出“从收藏构建”或“使用 AI 推荐”的两种起点。
- **[推断]** 模板和内容能降低冷启动，但热度不是可靠性；社区行程的日期、营业信息和交通必须重新核实。

Juzi 可以提供“路线骨架变体”，例如亲子、轻松、美食、暴走四种节奏，但每次生成仍要重新核实事实，不能复制旧攻略的易变数据。

## 6. 数据需求与刷新策略

### 6.1 数据域

| 数据域 | 关键字段 | 推荐来源顺序 | 时效策略 | 失败处理 |
|---|---|---|---|---|
| 行程身份 | 出发地、目的地、绝对日期、人数、偏好、体力约束 | 用户输入 | 创建时确认 | 显示假设，不静默猜测关键条件 |
| 城际交通 | 班次、站点、出发/到达、耗时、票价、余票状态 | 运营方/12306/航司 | 用户主动刷新；下单前复核 | 保留静态方案，打开官方预填查询 |
| 市内路线 | 坐标、距离、预计时长、交通方式、换乘 | 官方地图/开放路线服务 | 路线变更时重算 | 给出地图入口和人工顺序，不让空地图覆盖正文 |
| 船票/门票/预约 | 预售期、时间段、入口、证件、退改 | 运营方/景区/政府 | 每次出发前复核 | 明确替代路线和触发条件 |
| 天气与预警 | 日/小时温度、降雨、风、体感、预警 | 气象官方/Open-Meteo | 出发前每日；途中按需刷新 | 显示最近快照、核实时间和官方预警入口 |
| 住宿 | 每晚/总价、税费、房型、取消、位置、噪声、适合人群 | 酒店官网、主流平台 | 生成时快照；下单前复核 | 使用价格区间，不承诺库存 |
| POI | 名称、类别、坐标、开放时间、门票、建议时长、官方链接 | 官方、政府、开放地图、可靠内容源 | 稳定字段低频；开放时间出发前复核 | 标记未核实字段，不用评分代替事实 |
| 餐饮 | 门店、地址、营业时间、价格、必点、排队、分店歧义 | 品牌/指南/政府/地图交叉 | 生成时与出发前复核 | 提供同片区备选，避免单店依赖 |
| 预算 | 分类、单价、人数口径、总价、币种、已付/待付 | 上述数据汇总 + 用户记录 | 输入变化时重算 | 展示范围与不包含项 |
| 来源与信任 | URL、来源类型、核实时间、快照状态、置信说明 | 每个事实的原始来源 | 与事实一起更新 | 缺来源的易变事实不得写成确定结论 |

### 6.2 刷新分级

- **高频、用户主动刷新**：余票、航班状态、船班、临时闭园、气象预警。刷新失败必须保留正文和官方入口。
- **中频、生成时快照**：酒店价格、门票价格、天气预报、热门时段。显示核实时间和价格口径。
- **低频、定期复核**：POI 描述、地址、常规开放规则、线路关系。出发前仍需做最后复核。
- **本地持久数据**：清单、已订状态、个人备注、实际支出。只保存在当前设备，不上传、不暗示跨设备同步。

### 6.3 数据展示纪律

1. 每个易变数字必须回答“来源是什么、何时核实、是否含税、对几个人、还能否直接预订”。
2. 推荐和事实分开：`酒店离码头近` 是可核实事实，`综合首选` 是基于偏好的判断。
3. 评分、浏览量和复制量只是热度信号，不能替代安全、营业、价格和交通核实。
4. AI 生成的顺序必须经过营业时间、预约时间、路程和体力约束校验。
5. 离线页面保留最后快照，并明确“快照不是实时库存”。

## 7. 交互模式库

### 7.1 适合近期实现

1. **时间线与地图双向定位**：点击站点高亮地图；点击地图点滚动到时间节点。保持原版视觉，只增加状态样式。
2. **日期分段控制**：移动端以 `DAY 1 / DAY 2 / DAY 3` 切换行程与地图，选择状态跨视图保留。
3. **下一站动作条**：显示出发时间、预计路程、票证提醒和“打开地图”，旅行结束后自动转为返程提醒。
4. **显式刷新状态**：按钮文案区分“未刷新、刷新中、刚刚核实、刷新失败”；失败后保留快照。
5. **轻量比较控制**：交通和酒店支持按“省钱 / 省时 / 少折腾”切换推荐说明，不隐藏完整清单。
6. **可撤销建议**：系统建议调整顺序时先展示原因和影响，由用户确认；提供撤销，不自动重排。
7. **本地状态**：清单、已订和备注使用本地存储；页面明确只在当前设备保存。

### 7.2 适合后续验证

1. **拖拽排程**：只有在能同步校验营业时间、预约和路程时才有价值；否则只是制造错误顺序。
2. **路线自动优化**：应给出“少走路 / 少换乘 / 更省钱”目标，并允许锁定不能移动的预约。
3. **同行协作**：单文件交付天然不适合实时多人编辑；优先做可分享 PDF/HTML 和备注导出。
4. **AI 对话改行程**：必须显示改动摘要、受影响日期、预算和预订风险，不能直接覆盖原安排。

### 7.3 无障碍与操作细节

- 移动触控目标至少 `44 x 44px`，不能靠小字链接承担关键操作。
- 地图不是唯一入口；所有站点和路线都要有正文与外部导航链接。
- 颜色不能是状态的唯一表达，加载/成功/失败都要有文字。
- 支持键盘、焦点可见、`prefers-reduced-motion`，切换日期时不抢焦点。
- 任何弹窗都不能在用户理解价值前遮住核心内容。

## 8. 移动端原则

竞品反复显示出一个清楚的设备分工：**电脑负责比较和编排，手机负责执行和应变**。

Juzi 移动端应优先显示：

1. 当前核实时间和当日风险；
2. 下一站、出发时间、路程和地图入口；
3. 今天的时间线和票证/预约状态；
4. 天气、备选方案和返程提醒；
5. 其余酒店、餐厅和来源详情按需展开。

不应把桌面双栏等比例缩小。地图和时间线改为同一状态下的两个视图；底部导航只保留高频入口，不能让十多个标签挤成横向滚动的文字墙。

## 9. Juzi 当前优势与差距

### 9.1 已有优势

- 厦门原版首屏以目的地、日期、车次、船票、住宿区和核心预算建立了很强的旅行记忆点。
- 内容不是通用景点清单，而是按真实交通、午休、天气、预约和体力组织的逐时路线。
- 高铁、轮渡、酒店、美食、预算、天气、备忘和来源都在一个离线文件中，不依赖账号体系。
- 官方入口、核实时间、恶劣天气备选和“不要做什么”已经比多数只做灵感推荐的产品更可执行。
- 迭代版已修复关键移动端溢出、触控尺寸和天气失败状态，同时保持桌面原版像素一致。

### 9.2 真正差距

| 差距 | 影响 | 竞品参照 | 建议 |
|---|---|---|---|
| 时间线与地图是相邻模块但状态不联动 | 用户要自己对照站点与空间 | Wanderlog、Tripomatic | `P0/P1` 建立日期和站点共享状态 |
| 交通/酒店推荐多，但缺少紧凑比较 | 难快速理解“为什么是它” | Google Travel | 加省时/省钱/少折腾三种比较口径 |
| 已订、待订、凭证分散 | 途中容易找不到下一动作 | TripIt | 统一成本地“行程状态/凭证”摘要 |
| 实时性表达不统一 | 用户难判断哪些是快照 | Google Travel、TripIt | 统一核实时间、来源、快照/实时状态组件 |
| 旅行后无法记录实际支出或备注 | 难复盘和复用 | Wanderlog | 放到 `P2`，不阻塞行前/途中主流程 |
| 单文件不支持实时协作 | 同行人难共同编辑 | Wanderlog、Tripadvisor | 先优化分享与打印，不急建账号后端 |

## 10. 增量迭代路线

### P0：先把“拿着就能走”做扎实

1. 保持原版首屏、配色、章节顺序和可见文案，不进行整站重设计。
2. 把日期、站点和地图高亮做成共享状态；先实现点击定位，不上拖拽。
3. 统一实时模块状态：`快照时间 / 正在刷新 / 刷新成功 / 失败仍可读 / 官方复核`。
4. 在移动端加入轻量“今天 / 下一站”动作条，但不遮挡正文和底部导航。
5. 对交通、船票、预约和天气增加明确的风险触发条件与备选入口。

**验收**：桌面仍与指定原版一致；`390px` 和 `320px` 无横向溢出；无网时正文、行程、预算、来源和地图入口仍可用。

### P1：提高比较和调整效率

1. 交通增加“主选 / 备选”紧凑对比，字段统一为总价、耗时、站点、余量和风险。
2. 酒店增加“省钱 / 少折腾 / 海景升级”分段控制，只改变排序和推荐理由，不删原卡片。
3. 行程站点增加预计路程和交通方式；点击地图点可回到对应时间节点。
4. 增加可打印行程摘要，包含日期、车次、酒店、预约、紧急备选和官方入口。

**验收**：所有切换可键盘操作，刷新失败和地图不可用时有可读兜底，用户改选后预算同步更新。

### P2：只在验证需求后增加复杂能力

1. 可锁定预约的拖拽排程与撤销；
2. 按“少走路 / 少换乘 / 省钱”给出路线优化建议；
3. 本地实际支出与备注；
4. 导出变更摘要，供同行人查看；
5. AI 调整前后差异预览，不直接覆盖原行程。

**验收**：任意自动调整都不能破坏固定车次、预约、营业时间和返程缓冲；用户能看到原因、影响和撤销入口。

## 11. 不照搬清单

1. **不照搬 Roadtrippers 的首屏强制试用弹窗**：本次首屏被免费试用表单遮挡，用户还没理解产品就被索取邮箱。
2. **不照搬携程的全业务导航密度**：对单份攻略而言，酒店、金融、礼品卡等入口会稀释“下一步做什么”。
3. **不照搬穷游公共行程的直接复制**：本次列表存在远期或异常日期；任何模板都必须重新核实。
4. **不照搬登录墙**：Tripadvisor 的行程核心功能登录后才可用，而 Juzi 的价值之一就是离线直接打开。
5. **不把 AI 作为唯一入口**：保留手动查看、官方链接和静态正文，AI 只给可解释建议。
6. **不内嵌不稳定票务私有接口**：Google Travel 也明确要求向服务商复核；Juzi 继续使用快照 + 官方查询入口。
7. **不把桌面双栏硬缩到手机**：移动端切换视图，保留状态，不制造横向滚动。
8. **不为了“像 App”破坏原版叙事**：原版的视觉、章节和内容密度是基线，新增能力放在现有流程内部或之后。
