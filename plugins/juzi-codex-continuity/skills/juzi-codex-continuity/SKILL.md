# Juzi-Codex-Continuity

## 目标

让 Codex 在上下文压缩、换会话或中断后直接恢复正确的下一步，而不是重新泛化规划或重复搜索已经确认的结论。

本 skill 的唯一持久状态是目标项目内的 `.codex/ACTIVE_TASK.md`。如文件不存在，按本插件的 `templates/ACTIVE_TASK.md` 创建。不要创建数据库、索引、后台服务、MCP 服务器、额外的长期记忆文件或自动化钩子。

## 事实优先级

从高到低：

1. 当前 `git diff`、真实文件内容和本次测试结果。
2. `.codex/ACTIVE_TASK.md`。
3. 当前线程中用户显式创建的 `/goal`。
4. `AGENTS.md` 中的稳定规则与项目文档。
5. 聊天记录或压缩摘要。

当低优先级内容与高优先级证据冲突时，先更新状态文件，再继续执行。绝不能为了“无缝恢复”而忽略真实工作区。

## 启动或恢复

1. 若当前线程已有用户显式创建的 `/goal`，读取其目标和状态；没有 Goal 时不得为恢复方便而创建。
2. 读取 `.codex/ACTIVE_TASK.md`；文件不存在时，根据当前用户任务创建它，并在有活跃 Goal 时填写“关联 `/goal`”。
3. 运行 `git status --short`、检查与当前任务有关的 `git diff`，并读取状态文件中的最近测试证据。
4. 若 `NEXT_ACTION` 与真实状态一致，直接执行它。不要先重复摸底、重做计划或泛化设计。
5. 若状态过期、缺少证据或与工作区冲突，先在状态文件的“未完成 / 阻塞”中写清冲突，再做最小必要调查。

## 检查点

在以下事件后更新 `ACTIVE_TASK.md`：

- 用户确认目标、范围或验收标准。
- 用户创建、更新、完成或阻塞一个 `/goal`。
- 做出会影响后续实现的决定，或排除一个可行方案。
- 编辑完成一个可验证的改动。
- 运行相关测试、发现失败或解决阻塞。
- 即将压缩、切换会话、等待用户复测或交接任务。

每次更新必须保持简洁：只记录当前任务恢复所需的事实、文件路径、命令和结果。不要写入密钥、完整终端日志、可从 Git 直接恢复的逐行 diff，或未经验证的猜测。

## NEXT_ACTION 规则

`NEXT_ACTION` 必须恰好有一个可执行动作，包含必要时的目标文件或命令。例如：

`修改 src/issues/stats.ts 的 backlog 统计，使 fixed 计入 backlog；随后运行 npm run test:golden。`

不允许使用“继续处理”“检查一下”“梳理链路”等不可验收表述。遇到阻塞时，`NEXT_ACTION` 写为解除阻塞所需的最小调查或用户确认。

## `/goal` 协作规则

- `/goal` 只保存最终结果和验收条件；`ACTIVE_TASK.md` 保存当前事实、改动、测试证据和唯一下一步。不要把实施清单或终端日志写进 Goal。
- 只有用户明确使用 `/goal` 或要求创建长期目标时，才调用 Goal 创建工具；不要根据任务规模自动创建、替换或结束 Goal。
- 涉及文件修改、命令执行、浏览器操作或外部副作用的 Goal，先验证状态文件可读写且至少一条只读命令能运行，再调用 Goal 创建工具；若环境不可用，报告阻塞但不要创建无法推进的 Goal。仅用于阅读、规划或方案比较的只读 Goal 可直接创建。
- CLI 会话头部已声明 `sandbox: read-only` 且 Goal 需要变更时，直接报告需要可写沙箱；不要先通过 shell 读取 Skill、记忆、项目或状态文件来重复证明该限制。
- 活跃 Goal 存在时，状态文件必须填写“关联 `/goal`”的状态、目标和关联说明；每次 `NEXT_ACTION` 都要直接推进该 Goal。
- 全部验收通过后，先在状态文件记录验证证据，再将 Goal 标为 `complete`。同一外部阻塞连续三次无法推进时，才可将 Goal 标为 `blocked`。

## 团队单写者租约

- 只有用户明确要求团队模式、委派或并行工作时，主 Agent 才能派出子 Agent。团队模式不是 Goal 的自动副作用。
- 派发前，主 Agent 必须在 `ACTIVE_TASK.md` 的“团队协作与文件所有权”栏目登记模式、每个 Agent 的允许文件、禁止文件、只读范围、验收命令和租约释放条件。
- 同一文件、共享配置、接口契约或迁移脚本在任一时刻只能有一个写者。写入 Agent 持有租约期间，主 Agent 和其他 Agent 不得修改该范围；侦察 Agent 必须只读，不得调用编辑工具。
- 若 Agent 发现需要额外文件，必须停止在边界并请求主 Agent 分配。主 Agent 验收其实际 `git diff --name-only`、定向测试和范围后，才可释放原租约并授予下一租约。
- 未认领路径出现改动、用户同时编辑租约路径、或共享接口发生冲突时，立即暂停相关写入，读取真实 diff 后由主 Agent 串行重新分配；不得覆盖、回退或假定改动来源。
- 子 Agent 不得修改 `ACTIVE_TASK.md`、Goal 状态、共享配置或提交历史，除非主 Agent 明确把该单一文件的写入租约授予它。

## 压缩交接

需要准备压缩摘要时，使用本插件的 `templates/compact-prompt.md`。摘要必须覆盖状态文件的九个栏目，尤其不能丢失“关联 `/goal`”、“团队协作与文件所有权”和 `NEXT_ACTION`。

压缩后，仍以“启动或恢复”流程校验真实工作区；压缩摘要不是事实来源。

## 完成条件

在声称任务完成前：

1. 运行与改动相关的最小验证。
2. 在状态文件中记录验证命令和结果。
3. 确认没有未说明的 blocker，且 `NEXT_ACTION` 标记为“无，等待用户验收”或已明确下一阶段入口。
