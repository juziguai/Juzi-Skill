# Juzi-Codex-Continuity

## 目标

让 Codex 在上下文压缩、换会话或中断后直接恢复正确的下一步，而不是重新泛化规划或重复搜索已经确认的结论。

本 skill 的唯一持久状态是目标项目内的 `.codex/ACTIVE_TASK.md`。如文件不存在，按本插件的 `templates/ACTIVE_TASK.md` 创建。不要创建数据库、索引、后台服务、MCP 服务器、额外的长期记忆文件或自动化钩子。

## 触发判定

只根据用户请求和当前上下文进行一次入口判断，不得先调查仓库再决定。满足任一条件就触发：

- 用户明确要求使用本 skill、恢复长期任务或维护 `ACTIVE_TASK.md`。
- 任务明显跨多个实施/验证阶段，可能跨上下文压缩、会话、中断或等待用户复测。
- 任务涉及生产基线、破坏性动作预算、rollback/rescue，或需要团队文件租约。
- 当前项目已有属于同一任务的未完成 `ACTIVE_TASK.md`，需要继续恢复。

普通问答、解释、单命令只读查询、明确的一步验收和可在当前回合闭环的单文件小改默认不触发，除非用户明确要求或涉及生产保护。判断标准不是任务“看起来重要”，而是中断后是否确实需要依赖项目文件恢复执行状态。

## 初始化门禁（必须最先执行）

新的顶层任务到达时，先仅根据用户请求和当前上下文判断是否命中本 skill；不得为了完成这个判断先搜索仓库、运行 `git status`、制定实施计划或派出 Agent。

一旦命中本 skill，创建或读取目标项目的 `.codex/ACTIVE_TASK.md` 必须成为第一个任务侧动作；若文件需要新建，新建它必须是第一个持久化动作：

1. 只允许先读取本 skill、确认目标项目根目录，以及检查状态文件是否存在和可写。
2. 文件不存在时，立即按模板创建，并至少填写目标与验收、任务状态、初始化状态/时间/触发原因、当前范围与禁区、已知阻塞和唯一 `NEXT_ACTION`。
3. 文件存在时，立即读取并确认它属于当前任务；同一任务按“启动或恢复”继续。若另一任务状态为 `complete`，可用当前任务接管并重写；若另一任务仍为 `active`、`waiting_user` 或 `blocked`，必须暂停并请求用户确认是否切换，不得静默覆盖。
4. 状态文件不可创建、不可读取或不可写时，立即报告阻塞并停止任务执行；不得先继续工作、执行到中途再补建。
5. 初始化门禁完成后，才允许读取 `/goal`、运行 `git status`、搜索源码、制定计划、派 Agent、编辑文件、运行测试或执行外部动作。

如果由于触发遗漏而在执行中途才发现需要本 skill，必须立即停止后续动作、补建状态并如实标记“迟建”及已完成事项；迟建只能用于救援，不能视为满足初始化门禁。

## 权限与事实分层

不要把“允许做什么”和“当前真实状态是什么”混成同一优先级。

操作权限与范围以当前用户明确指令为准；`ACTIVE_TASK.md`、`/goal`、项目文档和历史摘要只能记录或解释范围，不能自行扩大权限。状态文件和 Goal 都不是 commit、push、发布、删除、覆盖、重启或其他破坏性动作的授权凭证。

事实证据从高到低为：

1. 当前 `git diff`、真实文件内容、本次测试结果和重新查询得到的生产实时状态。
2. `.codex/ACTIVE_TASK.md` 中有证据入口的已确认事实。
3. 当前线程中用户显式创建的 `/goal` 所记录的目标与验收。
4. `AGENTS.md` 中的稳定规则与项目文档。
5. 历史聊天记录或压缩摘要。

当前用户指令与状态文件范围不一致时，先按用户的新指令更新状态文件；用户对事实的假设与实时证据冲突时，明确报告证据差异。低优先级事实与高优先级证据冲突时，先修正状态文件再继续执行，绝不能为了“无缝恢复”而忽略真实工作区。

## 启动或恢复

1. 先确认“初始化门禁”已经完成；未完成时立即创建或读取 `.codex/ACTIVE_TASK.md`，不得执行后续步骤。
2. 若当前线程已有用户显式创建的 `/goal`，读取其目标和状态并更新状态文件；没有 Goal 时不得为恢复方便而创建。
3. 运行 `git status --short`、检查与当前任务有关的 `git diff`，并读取状态文件中的最近测试证据。
4. 若“生产状态与破坏性边界”存在，先以只读查询确认受保护基线、动作预算和 rollback/rescue 仍有效；PID、端口和健康快照不得沿用旧摘要。
5. 若状态与真实证据一致，按 `任务状态` 分流，不得把所有 `NEXT_ACTION` 都当成 Agent 动作：
   - `active`：直接执行唯一动作，不重复摸底、重做计划或泛化设计。
   - `waiting_user`：保持现场并等待已记录的用户动作；条件未变化时不重复追问。
   - `blocked`：只执行已记录的最小解阻检查；阻塞条件未变化时不得假装继续推进。
   - `complete`：不得执行 `NONE`；等待新任务，并按初始化门禁决定是否接管状态文件。
6. 若状态过期、状态与 `NEXT_ACTION` 不匹配、缺少证据或与工作区冲突，先在状态文件的“未完成 / 阻塞”中写清冲突，再做最小必要调查。

## 任务状态机

`任务状态` 只能取以下四个值，并决定 `NEXT_ACTION` 的合法形式：

| 状态 | 含义 | `NEXT_ACTION` |
|---|---|---|
| `active` | Agent 可继续推进 | 恰好一个由 Agent 执行的动作 |
| `waiting_user` | 需要用户验收、选择或补充信息 | 恰好一个用户动作，并写明收到后如何继续 |
| `blocked` | 外部条件或环境阻止继续 | 恰好一个最小解阻动作或所需状态变化 |
| `complete` | 验收条件已满足且无遗留动作 | 必须且只能写 `NONE` |

状态转换必须由证据驱动：开始或恢复为 `active`；只有确实需要用户输入时转为 `waiting_user`；同一外部阻塞连续三次仍无法推进时才转为 `blocked`；验证通过但尚待用户验收时仍是 `waiting_user`；用户验收或无需用户验收且全部条件满足后才转为 `complete`。`任务状态` 与 `/goal` 状态相互独立，不得用一个代替另一个。

## 检查点

在以下事件后更新 `ACTIVE_TASK.md`：

- 用户确认目标、范围或验收标准。
- 用户创建、更新、完成或阻塞一个 `/goal`。
- 做出会影响后续实现的决定，或排除一个可行方案。
- 编辑完成一个可验证的改动。
- 运行相关测试、发现失败或解决阻塞。
- 用户手动恢复服务、Runtime/PID 发生变化，或即将/刚刚执行 stop、restart、kill、切换、清理等破坏性动作。
- 即将压缩、切换会话、等待用户复测或交接任务。

## 字段更新矩阵

“必须更新”表示该事件发生后必须重新确认字段仍准确，并在值发生变化时立即写入；不得为了制造更新时间而重复写入相同内容。每个检查点都要复核控制三元组：`任务状态`、`未完成 / 阻塞`、`NEXT_ACTION`，但只在事件改变其含义时修改。

首次创建或用新任务接管一个 `complete` 文件时，十个顶级栏目必须全部显式初始化，不能保留空白占位：不适用的栏目写“无”“未建立”“单主 Agent”或“生产交互：否”；`任务状态` 初始为 `active`，`NEXT_ACTION` 填一个 Agent 可执行动作。

| 事件 | 必须更新 | 条件更新 | 不应更新 |
|---|---|---|---|
| 用户改变目标、验收或范围 | 目标与验收、当前范围与禁区、未完成项、控制三元组 | `/goal` 关联（目标受影响时） | 初始化时间、无关测试与生产记录 |
| 做出关键决定或排除方案 | 已证实事实与关键决定、未完成项、`NEXT_ACTION` | 范围、任务状态（决定改变执行条件时） | 无关历史事实和完整讨论过程 |
| 完成实际文件/配置改动 | 已做改动、未完成项、控制三元组 | 测试证据（仅标记因此失效或待验证的证据） | `/goal`、团队、生产栏目（未受影响时） |
| 运行测试、检查或实时复验 | 测试证据、未完成 / 阻塞、控制三元组 | 已证实事实、生产边界（结果改变它们时） | 已做改动（没有新改动时） |
| 等待用户选择、补充或验收 | `任务状态=waiting_user`、一个用户动作形式的 `NEXT_ACTION` | 未完成 / 阻塞（确有未闭环事项时） | 已确认事实、无关测试和改动记录 |
| 出现或解除外部阻塞 | 未完成 / 阻塞、任务状态、最小解阻 `NEXT_ACTION` | 生产边界、测试证据（与阻塞直接相关时） | 初始化信息和无关栏目 |
| 用户创建、修改、完成或阻塞 `/goal` | 关联 `/goal` | 任务状态、验收、`NEXT_ACTION`（Goal 改变当前执行时） | 已做改动和测试证据（事实未变化时） |
| 团队模式、租约或文件所有权变化 | 团队协作与文件所有权 | 当前范围与禁区、`NEXT_ACTION`（写入范围变化时） | Goal、生产与测试栏目（未受影响时） |
| 生产基线、授权、Runtime/PID 或破坏性动作变化 | 生产状态与破坏性边界；动作前后都更新 | 已证实事实、测试证据、未完成项、控制三元组 | 无关源码历史；不得沿用旧实时快照 |
| 压缩、换会话、等待复测或交接 | 重新校验全部十个栏目，修正已过期字段 | 只写发生变化的字段 | 不复制聊天过程或追加完整日志 |
| 完成任务 | 最终测试证据、未完成 / 阻塞=`无`、`任务状态=complete`、`NEXT_ACTION=NONE` | 生产边界（生产任务必须最终复验） | 不再制造新的实施动作 |

初始化状态、初始化时间和触发原因在创建后保持不变，只有真实发生“迟建”或新任务“接管”时才能改。`已做改动` 只记录实际发生的文件、配置或外部状态变化；`测试证据` 只记录真实运行或被明确判定失效的验证；未启用 Goal、团队或生产流程时，保留明确默认值，不反复刷新。

每次更新必须保持简洁：只记录当前任务恢复所需的事实、文件路径、命令和结果。状态文件不是追加式日志；更新时应删除已失效或被新证据取代的内容，只保留最新有效结论、未解决失败和恢复所需证据。默认控制在 200 行以内，超出时先压缩再继续，但不得删除未解决 blocker、生产保护边界、回滚救援信息或唯一 `NEXT_ACTION`。

不要写入密钥、认证头、密码、完整终端日志、可从 Git 直接恢复的逐行 diff，或未经验证的猜测。不得擅自修改 `.gitignore`、暂存或提交 `ACTIVE_TASK.md`；沿用项目现有 Git 策略，若文件已被跟踪则在汇报中明确说明。

## 生产状态与破坏性边界

任务可能触碰真实服务、数据、端口、Runtime 或系统配置时，状态文件必须增加并持续更新“生产状态与破坏性边界”：

- 是否接触生产，以及最后一次只读核验的时间和证据入口。
- 受保护基线：服务、父子进程/PID、端口、数据、配置和用户手动恢复状态。
- 允许动作与禁止动作；`stop`、`restart`、`kill`、删除、覆盖、切换和 cleanup 一律按破坏性动作记录。
- 本次授权对应的破坏性动作预算、已消耗次数和结果。失败、取消或 UAC 拒绝同样消耗单次预算。
- standby 就绪证据、停止条件、rollback 和 rescue 路径。

替代者未就绪前，不得把释放当前基线写成 `NEXT_ACTION`。用户手动恢复后立即冻结该 Runtime/PID；在可靠替代方案形成前，`NEXT_ACTION` 只能是只读复验或隔离修复。状态与实时证据冲突时，先保护服务并修正状态，不能按旧摘要继续执行。

## NEXT_ACTION 规则

`active`、`waiting_user` 或 `blocked` 状态下，`NEXT_ACTION` 必须恰好有一个可验收动作，包含责任方以及必要时的目标文件、命令或解除条件。例如：

`修改 src/issues/stats.ts 的 backlog 统计，使 fixed 计入 backlog；随后运行 npm run test:golden。`

不允许使用“继续处理”“检查一下”“梳理链路”等不可验收表述。`complete` 状态下不得伪造下一步，必须且只能写 `NONE`。

## `/goal` 协作规则

- `/goal` 只保存最终结果和验收条件；`ACTIVE_TASK.md` 保存当前事实、改动、测试证据和唯一下一步。不要把实施清单或终端日志写进 Goal。
- Agent 可以根据任务是否长期、多阶段、跨会话或具有长期验收周期，自行判断“建议建立 `/goal`”，把建议和理由写入状态文件，并在自然检查点向用户确认一次；条件未变化时不得反复询问。
- 建议不等于授权。只有用户明确使用 `/goal` 或明确要求创建长期目标时，才调用 Goal 创建工具；不得根据任务规模自动创建或替换 Goal。
- 涉及文件修改、命令执行、浏览器操作或外部副作用的 Goal，先验证状态文件可读写且至少一条只读命令能运行，再调用 Goal 创建工具；若环境不可用，报告阻塞但不要创建无法推进的 Goal。仅用于阅读、规划或方案比较的只读 Goal 可直接创建。
- CLI 会话头部已声明 `sandbox: read-only` 且 Goal 需要变更时，直接报告需要可写沙箱；不要先通过 shell 读取 Skill、记忆、项目或状态文件来重复证明该限制。
- 活跃 Goal 存在时，状态文件必须填写“关联 `/goal`”的状态、目标和关联说明；每次 `NEXT_ACTION` 都要直接推进该 Goal。
- 全部验收通过后，先在状态文件记录验证证据，再将 Goal 标为 `complete`。同一外部阻塞连续三次无法推进时，才可将 Goal 标为 `blocked`。

## 团队单写者租约

- 只有用户明确要求团队模式、委派或并行工作时，主 Agent 才能派出子 Agent。团队模式不是 Goal 的自动副作用。
- 派发前，主 Agent 必须在 `ACTIVE_TASK.md` 的“团队协作与文件所有权”栏目登记模式、每个 Agent 的允许文件、禁止文件、只读范围、验收命令和租约释放条件。
- 同一文件、共享配置、接口契约或迁移脚本在任一时刻只能有一个写者。写入 Agent 持有租约期间，主 Agent 和其他 Agent 不得修改该范围；侦察 Agent 必须只读，不得调用编辑工具。
- 若 Agent 发现需要额外文件，必须停止在边界并请求主 Agent 分配。主 Agent 验收其实际 `git diff --name-only`、定向测试和范围后，才可释放原租约并授予下一租约。
- 未认领路径出现改动、用户同时编辑租约路径、或共享接口发生冲突时，立即暂停相关写入，读取真实 diff 后由主 Agent 串行重新分配；不得覆盖、回退或假定改动来源。
- 子 Agent 不得修改 `ACTIVE_TASK.md`、Goal 状态、共享配置或提交历史，除非主 Agent 明确把该单一文件的写入租约授予它。

## 压缩交接

需要准备压缩摘要时，使用本插件的 `templates/compact-prompt.md`。摘要必须覆盖状态文件的十个栏目，尤其不能丢失“关联 `/goal`”、“团队协作与文件所有权”、“生产状态与破坏性边界”和 `NEXT_ACTION`。

压缩后，仍以“启动或恢复”流程校验真实工作区；压缩摘要不是事实来源。

## 完成条件

在声称任务完成前：

1. 运行与改动相关的最小验证。
2. 在状态文件中记录验证命令和结果。
3. 生产任务还要确认受保护基线未被遗留在不明状态、破坏性动作结果已真实复验、没有待执行或待救援请求。
4. 确认没有未说明的 blocker；尚待用户验收时设为 `waiting_user` 并填写一个用户验收动作，验收完成后设为 `complete` 且 `NEXT_ACTION` 写为 `NONE`。
