# Juzi-Codex-Continuity

一个面向 Codex 长任务的极简连续性插件，同时保存生产受保护基线、破坏性动作预算和 rollback/rescue 边界。

它不保存聊天原文，不使用数据库、MCP、向量检索、后台进程或 UI。它只让当前工作状态落在项目内的一个文件中，使上下文压缩、换会话或中断之后的 Codex 能从可验证的下一步继续。

## 唯一状态文件

在目标项目内维护 `.codex/ACTIVE_TASK.md`。文件模板见 [templates/ACTIVE_TASK.md](templates/ACTIVE_TASK.md)。

状态文件必须短小，只包含：

- 目标与验收标准
- 当前范围与禁区
- 已证实事实和关键决定
- 已做改动及测试证据
- 未完成项或阻塞项
- 唯一的 `NEXT_ACTION`

## 事实优先级

`git diff` 和测试结果高于 `ACTIVE_TASK.md`；`ACTIVE_TASK.md` 高于 `AGENTS.md` 中的任务性描述；压缩摘要只用于快速恢复，不能覆盖真实工作区。

## 使用方式

新的顶层任务到达时，先仅根据用户请求和当前上下文判断是否需要 `juzi-codex-continuity`。跨阶段、可能压缩/中断、等待复测、生产保护或团队租约任务应触发；普通问答、单命令查询和可在当前回合闭环的小改默认跳过。

一旦命中，创建或读取 `.codex/ACTIVE_TASK.md` 必须成为第一个任务侧动作；若文件需要新建，新建它必须是第一个持久化动作，并先于仓库摸底、`git status`、计划、委派、编辑、测试和外部动作。状态文件不可用时立即停止，不能执行到中途再补建。

初始化完成后，Codex 在关键决策、编辑完成、测试完成和恢复任务时继续更新或校验状态文件。

任务使用四态协议：`active`、`waiting_user`、`blocked`、`complete`。前三种状态必须保留一个明确的 `NEXT_ACTION`，完成态只能写 `NONE`，避免“已经完成却仍伪造下一步”。状态文件保存当前真相而非追加日志，默认控制在 200 行以内。

Codex 可以判断任务是否值得建立 `/goal` 并向用户建议，但只有用户明确授权后才能创建；自动连续性不扩大 Goal 权限。

权限和事实分别处理：当前用户明确指令决定允许范围，实时文件、diff、测试和生产查询决定真实状态；`ACTIVE_TASK.md` 与 `/goal` 都不能充当提交、发布、删除或重启授权。

字段更新采用最小矩阵：首次创建或接管时显式初始化全部十个栏目；后续检查点只更新事件直接影响的字段，并复核 `任务状态`、`未完成 / 阻塞`、`NEXT_ACTION` 三元组。未变化的初始化、Goal、团队、生产、测试和改动字段不得为“看起来更新过”而重复刷新。

压缩提示词模板见 [templates/compact-prompt.md](templates/compact-prompt.md)。是否把它写入全局 Codex 配置必须由用户明确确认。
